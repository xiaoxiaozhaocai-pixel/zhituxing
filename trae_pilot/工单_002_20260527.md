```typescript

    user_type: string,              // 来源：user_profiles.user_type（'free' / 'monthly' / 'semester' / 'yearly' / 'lifetime'）
    membership_type: string,         // 同上（向前兼容字段名）
```

1. **复用 `src/lib/quota.ts` 已有的查询函数**（如果 `getUserProfile` 或 `getUserQuotaFromDb` 是 export 的就直接 import；如果是模块内私有，把它们改成 export 再用，不要重新写一遍）

### 任务 2：notifications 三字段技术债清理 🟡 P1

**现状**：`notifications` 表当前有三个表达"已读"的字段并存：

- `is_read` BOOLEAN（最新统一字段）
- `read` BOOLEAN（历史字段）
- `status` VARCHAR（历史字段）

并且有一个 `BEFORE INSERT OR UPDATE` 触发器 `sync_read_status` 在三者间同步。

今天上午任务 2（commit `68de5e9`）已把代码层全部统一到 `is_read`，全项目 grep 已无 `read_at` 或裸 `read` 字段引用。**现在补一个 SQL migration 把表里的 `read`/`status` 字段和触发器删掉**。

**【生成两个文件】**

#### 文件 1：`sql/migrations/2026-05-27_notifications_drop_legacy.sql`

```sql

-- 前置：代码层已全部统一到 is_read（commit 68de5e9）
-- 执行者：人工在 Supabase Dashboard SQL Editor 执行
-- ⚠️ 执行前必须先备份：CREATE TABLE notifications_backup_20260527 AS SELECT * FROM public.notifications;

BEGIN;

-- 1. 删触发器
DROP TRIGGER IF EXISTS sync_read_status ON public.notifications;
DROP FUNCTION IF EXISTS public.sync_read_status();

-- 2. 删历史字段（保留 is_read）

ALTER TABLE public.notifications DROP COLUMN IF EXISTS status;

-- 3. 校验
SELECT column_name, data_type 
FROM information_schema.columns 
WHERE table_name = 'notifications' AND table_schema = 'public'
ORDER BY ordinal_position;
-- 期望：is_read 在列表里，read/status 不在

-- 如果执行后线上发现问题需要回滚，恢复方案：
```

## 附录：常见踩坑提醒

1. **AuthUser 字段是 `id` 不是 `userId`**（v1 翻车过）：`const userId = authResult.id`，不是 `authResult.userId`
2. **Supabase `.single()` vs `.maybeSingle()`**：用户记录可能不存在时用 `.maybeSingle()`（不抛错），强制存在时用 `.single()`
3. **SQL 字段名必须 grep 验证再写**（featured→is_featured 翻车教训）：写 SQL 前 `grep -rn "字段名" src/` 或 `cat buildfix_work/*_create.sql`
4. **pnpm build 跑不通别 push**：build 失败 push 上去 Zeabur 会构建失败影响生产
