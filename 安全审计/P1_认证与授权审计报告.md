### [P1-CRIT-1] 管理后台 AI 洞察 API 无认证检查

**严重程度**：严重（上线前必须修）  
**漏洞类型**：认证绕过  
**影响范围**：`/api/admin/dashboard/ai-insights`  
**文件位置**：`src/app/api/admin/dashboard/ai-insights/route.ts`

**攻击场景**：任何人无需登录即可访问管理后台的 AI 洞察数据，获取包括总用户数、新增用户趋势、会员统计、面试/评估数据、活跃公司排行、JD 提交统计等敏感运营数据。这是该管理模块唯一的无认证路由。

**当前代码**：
```typescript
// ❌ 完全没有认证检查！
export async function GET(request: NextRequest) {
  try {
    const supabase = getSupabaseAdmin();
    // 直接查询用户数据、JD数据、面试数据...
    const { count: totalUsers } = await supabase.from('users').select('*', { count: 'exact', head: true });
    // ...
```

**修复代码**：
```typescript
import { getAuthenticatedUserId } from '@/lib/auth';

async function checkAdmin(request: NextRequest): Promise<boolean> {
  const userId = await getAuthenticatedUserId(request);
  if (!userId) return false;
  const adminIds = process.env.ADMIN_USER_IDS;
  if (!adminIds) {
    console.warn('[admin/ai-insights] ADMIN_USER_IDS not configured');
    return false;
  }
  const list = adminIds.split(',').map((x) => x.trim().toLowerCase());
  return list.includes(userId.toLowerCase());
}

export async function GET(request: NextRequest) {
  if (!(await checkAdmin(request))) {
    return NextResponse.json({ code: 403, message: '无权限' }, { status: 403 });
  }
  // ... 原有逻辑
```

### [P1-CRIT-2] 认证方式不一致导致安全漏洞风险

**严重程度**：严重（上线前必须修）  
**漏洞类型**：认证绕过  
**影响范围**：`/api/user/profile`、`/api/subscriptions`、`/api/filters`  
**文件位置**：
- `src/app/api/user/profile/route.ts`
- `src/app/api/subscriptions/route.ts`
- `src/app/api/subscriptions/[id]/route.ts`

**攻击场景**：这些路由使用手动 Cookie 解析 + `createClient(SERVICE_ROLE_KEY)` 创建 Supabase 客户端来验证 token。问题在于：(1) 手动解析 Cookie 可能遗漏 edge cases；(2) 使用 SERVICE_ROLE_KEY 创建的客户端拥有数据库管理员权限，一旦 token 验证逻辑有缺陷，后果严重。与 `auth.ts` 中 `getAuthenticatedUser()` 的标准化认证不一致。

**当前代码**（`user/profile/route.ts`）：
```typescript
// ❌ 手动解析 Cookie，绕过统一认证函数
const cookieHeader = request.headers.get('cookie') || '';
const tokenMatch = cookieHeader.match(/sb-access-token=([^;]+)/);
const token = tokenMatch ? tokenMatch[1] : null;
// 使用 SERVICE_ROLE_KEY 创建客户端
const supabase = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!
);
```

**修复代码**：
```typescript
import { getAuthenticatedUser, getAuthenticatedUserId } from '@/lib/auth';
import { getSupabaseAdmin } from '@/lib/supabase';

export async function GET(request: NextRequest) {
  const userId = await getAuthenticatedUserId(request);
  if (!userId) {
    return NextResponse.json({ error: '未登录' }, { status: 401 });
  }
  
  const supabase = getSupabaseAdmin();
  const { data: profile } = await supabase
    .from('user_profiles')
    .select('*')
    .eq('user_id', userId)
    .maybeSingle();
  // ...
```

**修复验证**：
1. 确保所有路由使用统一的 `getAuthenticatedUser()` / `getAuthenticatedUserId()` 函数
2. 手动 Cookie 解析仅保留在 `auth.ts` 和 `auth-cookies.ts` 中
3. 所有认证逻辑在 `auth.ts` 中集中管理

### [P1-MED-1] RLS 策略依赖 SERVICE_ROLE_KEY 绕过，实际防护效果减弱

**严重程度**：中等（建议上线前修复）  
**漏洞类型**：纵深防御缺失  
**影响范围**：所有使用 `getSupabaseAdmin()` 的 API 路由（约 60+ 路由）  
**文件位置**：`src/lib/supabase.ts`、大量 API 路由文件

**风险描述**：项目大量使用 `getSupabaseAdmin()`（SERVICE_ROLE_KEY）操作数据库，这会完全绕过 RLS 策略。虽然 `p1_rls_fix.sql` 已准备，但实际防护效果被削弱。应该在直接面向用户的查询中使用 `getSupabase()`（ANON_KEY），让 RLS 真正生效。

**建议修复**：
1. 审计所有使用 `getSupabaseAdmin()` 的地方，区分：
   - 需要绕过 RLS 的管理操作 → 保留 `getSupabaseAdmin()`
   - 普通用户数据查询 → 改用 `getSupabase()` + 在请求中设置用户 JWT
2. 对于用户自己的数据查询，使用 `getSupabase()` 并配合 `auth.uid()` 的 RLS 策略

**示例修复**（以用户订单查询为例）：
```typescript
// 当前：使用 admin client 绕过 RLS
const supabase = getSupabaseAdmin();
const { data } = await supabase.from('pending_orders')
  .select('*').eq('user_id', user.id);

// 建议：使用 anon client + 设置 auth context
const supabase = getSupabase();
await supabase.auth.setSession({ access_token: token, refresh_token: '' });
const { data } = await supabase.from('pending_orders').select('*');
// RLS 策略自动过滤：auth.uid() = user_id
```

> ⚠️ 注意：`getSupabase()` 在当前代码中创建的是 ANON_KEY 客户端，配合 Supabase Auth 的 `setSession` 后，RLS 策略将生效。这是推荐的做法。

### [P1-MED-2] admin_token 明文存储于环境变量，缺乏轮换机制

**严重程度**：中等（建议上线前修复）  
**漏洞类型**：凭证管理  
**影响范围**：`/api/admin/auth/route.ts`、管理后台全模块  
**文件位置**：`src/app/api/admin/auth/route.ts`

**风险描述**：管理员 token（`ADMIN_TOKEN`）直接与环境变量比较，无哈希、无过期时间、无轮换机制。如果 token 泄露，攻击者可永久访问管理后台。

**当前代码**：
```typescript
if (adminToken && process.env.ADMIN_TOKEN && adminToken === process.env.ADMIN_TOKEN) {
  return NextResponse.json({ isAdmin: true, userId: 'admin', authMethod: 'admin_token' });
}
```

**建议修复**：
1. 将 `ADMIN_TOKEN` 改为哈希存储（如 bcrypt hash）
2. 添加 token 过期时间检查
3. 实现 token 轮换 API
4. 记录管理员登录日志

**修复代码**：
```typescript
import crypto from 'crypto';

function verifyAdminToken(token: string): boolean {
  const storedHash = process.env.ADMIN_TOKEN_HASH;
  if (!storedHash) return false;
  // 使用 timing-safe 比较防止时序攻击
  const hash = crypto.createHash('sha256').update(token + process.env.ADMIN_TOKEN_SALT).digest('hex');
  return crypto.timingSafeEqual(Buffer.from(hash), Buffer.from(storedHash));
}
```

### [P1-MED-3] 部分 API 路由使用直接 fetch + SERVICE_ROLE_KEY，暴露密钥在请求头中

**严重程度**：中等（建议上线前修复）  
**漏洞类型**：凭证暴露  
**影响范围**：`assessment/route.ts`、`chat/chat-history.ts`、`coze/sse.ts`、`coze/context.ts`  
**文件位置**：
- `src/app/api/assessment/route.ts`
- `src/app/api/chat/chat-history.ts`
- `src/lib/coze/sse.ts`
- `src/lib/coze/context.ts`

**风险描述**：这些代码片段使用 `fetch()` + `apikey`/`Authorization` 头直接携带 `SUPABASE_SERVICE_ROLE_KEY` 调用 Supabase REST API，而不是使用 Supabase JS 客户端。虽然这些代码仅在服务端执行，但增加了密钥通过日志/错误信息泄露的风险。

**当前代码**（`chat-history.ts`）：
```typescript
const res = await fetch(`${process.env.NEXT_PUBLIC_SUPABASE_URL}/rest/v1/chat_history`, {
  method: 'POST',
  headers: {
    'apikey': process.env.SUPABASE_SERVICE_ROLE_KEY || '',
    'Authorization': 'Bearer ' + (process.env.SUPABASE_SERVICE_ROLE_KEY || ''),
    'Content-Type': 'application/json',
    'Prefer': 'return=minimal',
  },
  body: JSON.stringify({ ... }),
});
```

**建议修复**：统一使用 `getSupabaseAdmin()` 客户端，避免手动拼接 HTTP 请求：
```typescript
const supabase = getSupabaseAdmin();
const { error } = await supabase.from('chat_history').insert({ ... });
```

### [P1-MED-4] CSP 配置包含 'unsafe-inline' 和 'unsafe-eval'，削弱 XSS 防护

**严重程度**：中等（尽快修复）  
**漏洞类型**：安全配置  
**影响范围**：全站  
**文件位置**：`src/lib/security-headers.ts`

**风险描述**：CSP 中的 `script-src 'self' 'unsafe-eval' 'unsafe-inline'` 允许内联脚本和 eval，大幅削弱 XSS 防护能力。攻击者若能注入任意 HTML，即可执行任意 JavaScript。

**当前代码**：
```typescript
const ContentSecurityPolicy = `
  default-src 'self';
  script-src 'self' 'unsafe-eval' 'unsafe-inline';
  style-src 'self' 'unsafe-inline';
  ...
```

**建议修复**：使用 nonce 或 hash 替代 `unsafe-inline`：
```typescript
// 方案：使用 CSP nonce（需要服务端渲染配合）
// middleware.ts
const nonce = crypto.randomUUID();
request.headers.set('x-csp-nonce', nonce);

// security-headers.ts
const ContentSecurityPolicy = `
  default-src 'self';
  script-src 'self' 'nonce-${nonce}';
  style-src 'self' 'unsafe-inline';
  ...
```
> 注：如果 Next.js 框架内部使用 `unsafe-eval`（如开发模式的热更新），至少应在生产环境移除 `'unsafe-inline'`。
